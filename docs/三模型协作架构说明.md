# DesignEcho 三模型协作架构说明

## 一、三个模型角色

| 角色 | 设置项 | 典型模型 | 职责 |
|------|--------|----------|------|
| **逻辑理解** | `layoutAnalysis` | Qwen2.5 32B、DeepSeek Coder、Gemini Pro | 排版分析、代码生成、任务规划、工具调用决策 |
| **文案优化** | `textOptimize` | Qwen2.5 14B、Gemini Flash | 营销文案、标题撰写、卖点提炼 |
| **视觉分析** | `visualAnalyze` | LLaVA、Gemini Flash、Qwen3-VL | 图像理解、设计分析、参考图对比 |

## 二、任务类型与模型路由

用户消息会先被分类为以下任务类型之一：

| 任务类型 | 触发关键词示例 | 使用的模型 |
|----------|----------------|------------|
| **visual** | 查看图片、分析设计、卖点分类、理解图片内容、参考图 | `visualAnalyze` |
| **logic** | 抠图、移动图层、居中、创建文档、SKU 批量 | `layoutAnalysis` |
| **copywriting** | 写文案、标题、卖点、广告语 | `textOptimize` |
| **general** | 低置信度或无法分类 | `layoutAnalysis`（通用能力最强） |

## 三、路由逻辑（useChatActions）

```
用户输入
  ↓
detectTaskTypeWithConfidence()
  ↓
taskType: 'visual' | 'logic' | 'copywriting' | 'general'
  ↓
getModelPriorityForTask(taskType)
  ↓
根据 mode (local/cloud/auto) 和 autoFallback
  ↓
模型优先级列表，如: [qwen2.5:32b] → [gemini-3-flash] (fallback)
  ↓
依次尝试，直到成功
```

## 四、analyzeAssetContent 的视觉模型

`analyzeAssetContent` 工具内部使用**独立的视觉模型链**，与主对话模型不同：

- **调用位置**：`resource-handlers.ts` → `resource:analyzeAsset`
- **模型顺序**：gemini-3-flash → ollama-llava:13b → ollama-llava:7b
- **作用**：读取图片 Base64，用视觉模型分析，返回结构化 JSON（分类、主体、颜色、建议用途）

主对话模型（可能是 qwen2.5:32b，无视觉）只需**正确调用** `analyzeAssetContent`，无需自己「看」图。

## 五、常见问题

### Q: 为什么「查看项目图片」会去打开 Photoshop？

**A**: 主模型误判为「需要创建文档」类任务，调用了 createDocument。已通过 Prompt 明确禁止，并增加 visual 任务检测。

### Q: 三个模型是串行还是并行？

**A**: 单轮对话只用**一个**主模型。`analyzeAssetContent` 在工具执行时内部调用视觉模型，对主模型透明。

### Q: 如何确认当前用的是哪个模型？

**A**: 查看终端日志 `[ModelPriority] 模式=xxx, 任务=xxx, 优先级=xxx`，或设置面板中的「逻辑理解/文案优化/视觉分析」配置。
