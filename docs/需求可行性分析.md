## 4. 技术难点清单

### 4.1 难点分级

| 级别 | 定义 | 解决周期 |
|------|------|----------|
| 🔴 **高** | 需要技术攻关或外部依赖 | 3-6 周 |
| 🟡 **中** | 方案明确但工作量较大 | 1-3 周 |
| 🟢 **低** | 直接可实现 | < 1 周 |

### 4.2 难点详解

#### 🔴 D1: 多模态图像理解

```
问题描述：
  需要理解产品图片的内容（是什么产品、什么款式、什么场景）

技术挑战：
  1. 本地部署多模态模型（CLIP/BLIP）体积大、推理慢
  2. 调用云端 API（GPT-4V）成本高、响应慢
  3. 类目特定识别需要微调

可选方案：
  A. CLIP-ViT-L/14 ONNX（约 1.7GB）本地部署
     优点：离线可用、无调用成本
     缺点：模型大、泛化能力有限

  B. 云端多模态 API（GPT-4V / Claude Vision / Qwen-VL）
     优点：能力强、无需部署
     缺点：成本高、网络依赖

  C. 混合方案：本地 CLIP 粗分类 + 云端 LLM 精细理解
     推荐度：⭐⭐⭐⭐⭐

建议：MVP 阶段先用云端 API，后期优化成本再考虑本地部署
```

#### 🔴 D2: 卖点/痛点知识库

```
问题描述：
  需要类目专属的卖点库和痛点库，支撑智能提炼

技术挑战：
  1. 知识库内容需要人工整理（袜子类目 100+ 卖点）
  2. 知识与素材的匹配逻辑
  3. 知识库的可扩展性（支持多类目）

可选方案：
  A. 结构化知识库（JSON/数据库）
     - 卖点表：{ id, name, keywords, description, priority }
     - 痛点表：{ id, problem, solution, related_sellpoints }
     
  B. 向量知识库（RAG 检索增强）
     - 使用 Embedding 模型将知识向量化
     - 根据产品图片描述检索相关知识

  C. LLM 内置知识 + 少量样例
     - Prompt 中嵌入类目知识
     - Few-shot 示例引导

建议：MVP 用方案 A（结构化），后期用方案 B（RAG）
```

#### 🔴 D3: 设计审美评估

```
问题描述：
  如何让 AI 具备"设计师级审美"，高于大众但不过于抽象

技术挑战：
  1. "审美"难以量化
  2. 缺乏电商设计质量评估数据集
  3. 主观性强，难以形成统一标准

可选方案：
  A. 规则化审美（可实现）
     - 定义明确的设计规范（间距、配色、对比度）
     - 检查设计是否符合规范
     - 输出"规范符合度"评分

  B. 参考优秀案例（可实现）
     - 收集 TOP 店铺设计案例
     - 提取视觉特征（配色比例、布局结构）
     - 作为设计决策参考

  C. 训练审美模型（难度高）
     - 收集设计稿 + 人工评分
     - 训练打分模型
     - 需要大量标注数据

建议：MVP 用方案 A（规则化），辅以方案 B（案例参考）
```

#### 🟡 D4: 模板智能适配

```
问题描述：
  不同产品尺寸、比例各异，如何智能适配到统一模板

技术挑战：
  1. 产品主体定位（已有 BiRefNet/SAM）
  2. 缩放比例计算
  3. 背景填充/延伸

解决方案：
  1. 使用现有抠图能力获取产品主体
  2. 计算主体包围盒
  3. 按模板占位区域缩放适配
  4. 背景使用纯色或智能填充

工作量：2 周
```

#### 🟡 D5: 颜色识别与聚类

```
问题描述：
  从产品图中准确提取 SKU 颜色，并生成规范的颜色名称

技术挑战：
  1. 背景色干扰
  2. 光照影响
  3. 颜色命名标准化（"藏青"vs"深蓝"）

解决方案：
  1. 先抠图去除背景
  2. 使用 K-Means 聚类提取主色
  3. 匹配预定义颜色名库（色差最小）

工作量：1-2 周
```

#### 🟡 D6: 详情页模块-素材匹配

```
问题描述：
  如何将素材图智能匹配到 14 个模块中的合适位置

技术挑战：
  1. 素材内容理解
  2. 模块需求定义
  3. 匹配算法

解决方案：
  1. 定义每个模块的"期望素材特征"
     - C 首屏：需要产品整体图
     - G 材质：需要面料特写
     - K 细节：需要局部特写
  2. 使用图像特征（构图、主体占比）匹配
  3. 支持用户手动调整

工作量：2-3 周
```

| 风险 | 影响 | 概率 | 缓解措施 |

```
│  ├── 详情页：5 个核心模块（非 14 个）                               │
│  └── 设计规范：固定配色/字体（非智能推荐）                           │
│                                                                     │
│  ❌ MVP 不做                                                         │
│  ├── 多模态智能理解（依赖 LLM）                                      │
│  ├── 卖点/痛点智能提炼                                              │
│  ├── 设计审美评估                                                   │
│  ├── 竞品分析                                                       │
│  └── 多类目支持                                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 7.2 技术攻关优先级

| 序号 | 技术难点 | 优先级 | 原因 |
|------|----------|--------|------|
| 1 | 模板系统 | P0 | 核心功能，决定设计输出质量 |
| 2 | SKU 批量生成 | P0 | 高频刚需，技术可行 |
| 3 | 颜色识别 | P1 | SKU 的前置依赖 |
| 4 | 素材分类 | P1 | 提升效率，可简化为规则 |
| 5 | LLM 接入 | P2 | 智能化核心，但可后期迭代 |
| 6 | 审美评估 | P3 | 锦上添花，非必须 |

### 7.3 知识库建设路径

```
Week 1-2: 袜子类目基础
├── 产品分类（中筒/船袜/长筒/隐形...）
├── 核心卖点 50 条（材质/功能/设计...）
├── 用户痛点 30 条（起球/臭脚/勒脚...）
└── 配色方案 10 套

Week 3-4: 模板库
├── 主图模板 3 套（点击图 + 转化图）
├── 详情页模板 2 套（标准 + 精简）
└── SKU 模板 2 套

Week 5-6: 规范与优化
├── 字体规范（标题/正文）
├── 排版规范（间距/对齐）
└── 质量检查规则
```
